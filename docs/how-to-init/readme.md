# How to Pre-Generate CLAUDE.md for Enterprise Claude Code Deployment

## Background

Claude Code's `/init` command generates a `CLAUDE.md` file by analyzing a codebase. This file is
loaded at the start of every Claude Code session to give the model project context. For enterprise
deployments, pre-generating `CLAUDE.md` before developers clone a repo saves onboarding time and
improves adoption consistency.

The problem: **`/init` is interactive-only and cannot be invoked programmatically.**

This document captures the workflow for replicating `/init` behavior via a document-agent, including
a gap analysis between the two approaches and the enhanced prompt that bridges them.

---

## The Two Approaches

### Approach 1: `/init` (interactive)

Running `/init` inside a Claude Code session analyzes the codebase and writes `CLAUDE.md`. It has
hardcoded awareness of what makes a good guidance file and infers CLI subcommands, module roles, and
architectural patterns by reading the code.

**Limitation:** Only works interactively. Cannot be scripted.

### Approach 2: Document-agent with a crafted prompt (`claude -p`)

A document-agent invokes the Claude Code CLI in non-interactive mode:

```bash
claude -p "Analyze this project and generate a CLAUDE.md file..." \
  --allowedTools "Read,Glob,Grep"
```

The prompt must be explicit about *what sections to include* and *what to look for*, since the
model has no hardcoded CLAUDE.md awareness in this mode.

**This is the officially recommended pattern.** The Claude Code docs state:
> *"In `-p` mode, describe the task you want to accomplish instead [of using slash commands]."*

Note: there is a `--init-only` flag, but it runs initialization hooks (pre-session shell commands
from config) — not CLAUDE.md generation. Different thing entirely.

---

## Why `/init` Cannot Be Run Programmatically

| Method | Works? | Notes |
|--------|--------|-------|
| `claude /init` (shell) | No | `/init` is only recognized inside an interactive session |
| `claude -p "/init"` | No | Slash commands are disabled in `-p` / `--print` mode |
| `--init-only` flag | No | Runs config hooks, not CLAUDE.md generation |
| `claude -p "<prompt>"` | **Yes** | Describe the task; model reads and writes the file |

## Gap Analysis: `/init` output vs. prompted output

Two `CLAUDE.md` files were generated for the SPL project and compared:

- `CLAUDE.md` — generated by `/init`
- `CLAUDE-prompted.md` — generated by a crafted prompt (first iteration)

### What `/init` had that the prompted version missed

| Gap | Detail |
|-----|--------|
| CLI subcommands | `spl validate`, `spl explain`, `spl execute`, `spl memory`, `spl rag` — absent entirely |
| Single-file test invocation | `python -m pytest tests/test_lexer.py` pattern not shown |
| Supporting modules | `functions.py`, `token_counter.py`, `explain.py`, `__init__.py` public API (`parse()`, `validate()`, `explain()`, `execute()`) |
| Optimizer budget algorithm | Step-by-step: reserve output → estimate per-item → apply LIMIT caps → compress proportionally → order by (cached → memory → RAG → context) |
| Extending adapters | "Add new providers by subclassing `spl/adapters/base.LLMAdapter`" — actionable instruction |
| Key design decisions | Why hand-written parser (research formalizeability), token budget philosophy, `.spl/` analogous to `.git/` |
| Runtime storage context | Files are created by `spl init`; explicit list of what gets created under `.spl/` |

**Root cause:** `/init` reads code and *infers* these things. A prompt-driven agent won't document
what it isn't explicitly instructed to look for.

### What the prompted version had that `/init` missed

| Addition | Value |
|----------|-------|
| Code style section | `from __future__ import annotations`, dataclasses, abstract base classes, YAML not JSON, case-insensitive keywords |
| Common gotchas section | The most operationally valuable part — hyphen tokenization, keyword-as-identifier, hash-based embeddings, FAISS/numpy conflicts |
| Per-test-file test counts | 14 / 20 / 6 breakdown makes it easy to spot regressions |
| Workflow section | When to update the arxiv paper, how to regenerate benchmark figures |

**The prompted version wins on actionable gotchas; the `/init` version wins on completeness of
commands and architecture depth.**

## The Enhanced Prompt

The gaps above were used to produce `CLAUDE-prompt.md` — the reusable document-agent prompt that
bridges both approaches. Key additions over the first-iteration prompt:

1. **Added `## CLI commands` section** — explicitly instructed the agent to find and list all `spl`
   subcommands. This was the single largest gap.
2. **Added `## Supporting modules` section** — `functions.py`, `token_counter.py`, `explain.py`,
   `__init__.py` public API.
3. **Added `## Key design decisions` section** — the "why" behind parser choice, token budget
   philosophy, adapter pattern, `.spl/` directory analogy.
4. **Enhanced Testing section** — added single-file test invocation pattern.
5. **Enhanced Architecture section** — optimizer budget algorithm steps, `base.LLMAdapter`
   subclassing instruction, PLY added to no-external-deps list, runtime storage file list.
6. **Enhanced Project overview** — added co-creation experiment context.

Sections kept as-is (already strong): code style, common gotchas, key files, workflow.

The prompt is saved at: `CLAUDE-prompt.md` in the project root.

| File | Role |
|------|------|
| `CLAUDE.md` | Generated by `/init` (interactive reference) |
| `CLAUDE-prompted.md` | Generated by first-iteration document-agent prompt |
| `CLAUDE-prompt.md` | Enhanced reusable prompt for document-agent (feed this to `claude -p`) |

## Usage

To pre-generate `CLAUDE.md` for a new project as part of enterprise deployment:

```bash
# From the project root
claude -p "$(cat /path/to/CLAUDE-prompt.md)" \
  --allowedTools "Read,Glob,Grep,Write"
```

Or integrate into your document-agent pipeline by passing the contents of `CLAUDE-prompt.md` as
the system or user prompt, with the working directory set to the target project root.
